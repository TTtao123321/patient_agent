# Router Agent 优化完成

## 实现总结

已成功优化 Router Agent，实现了基于 LLM 的意图识别系统。

### 核心功能

1. **LLM 意图识别**
   - 创建 `LLMClient` - 通过 Ollama 与 Qwen 模型交互
   - 优化 `IntentClassifier` - 支持 LLM 识别 + 关键词回退机制
   - 智能优先级排序确保高准确率

2. **支持四种意图类型**
   - `symptom_consult` - 症状咨询（咳嗽、发烧等）
   - `report_analysis` - 报告分析（血液检查、CT 等）
   - `medical_knowledge` - 医学知识（疾病治疗、用药建议等）
   - `record_query` - 病历查询（医疗历史、既往病史等）

3. **新增 RecordQueryAgent**
   - 处理医疗历史和病历查询
   - 支持多条历史记录的汇总和展示

### 测试过程中的思路

这次测试不是一次性跑完，而是按“先验证局部，再验证整体；先验证逻辑，再验证链路”的顺序逐步推进。核心思路如下。

#### 1. 先验证分类逻辑本身，再验证 HTTP 接口

第一步没有直接依赖接口返回结果，而是先在本地直接调用 `RouterAgent` 和 `IntentClassifier`。原因很简单：如果一开始就只看 HTTP 响应，无法判断问题究竟出在分类器、路由器、会话记忆，还是服务进程本身。

所以最先确认的是两件事：

- `IntentClassifier` 能否给出四类意图中的正确分类
- `RouterAgent.route()` 能否把分类结果映射到正确 Agent

这样做的价值是先把“分类逻辑正确性”和“接口链路正确性”拆开，避免一上来把多个变量混在一起排查。

#### 2. 再走 HTTP API，验证真实调用链路

本地逻辑通过后，才开始走 `/agent/chat` 接口测试。因为最终线上行为取决于完整链路，而不是单个类的局部结果。

接口侧重点主要有三项：

- 返回的 `intent` 是否正确
- 返回的 `agent_used` 是否与意图一致
- 会话上下文是否污染当前轮的分类结果

这一步的目的，是确认“分类正确”不只是类内部正确，而是在 FastAPI、SessionManager、RouterAgent 串起来以后仍然正确。

#### 3. 使用单例问题复现，而不是只看批量结果

在批量测试里，一旦出现错误，没有马上继续扩大样本，而是先拿失败样例单独复现。这样做是为了快速缩小排查范围。

例如测试过程中重点盯住了这几类句子：

- “我想了解一下我过去的医疗记录。”
- “感冒一般怎样治疗？”
- “我之前得过肺炎，现在又开始咳嗽了。”

这三个例子分别对应：

- 病历查询和报告查询的边界
- 医学知识和症状咨询的边界
- 既往病史和当前症状混合出现时的优先级问题

先做单条复现，再回到批量测试，是为了确认问题究竟是规则缺陷，还是测试上下文带来的偶发干扰。

#### 4. 明确区分“分类错误”与“服务状态异常”

测试过程中实际出现过两类完全不同的问题：

- 分类结果不符合预期
- HTTP 连接失败或服务无响应

这两类问题如果混在一起看，很容易误判成模型效果差。实际上，后者更多是服务进程状态、重启时机或测试脚本执行方式的问题。

因此在排查时专门把这两个方向拆开：

- 用本地脚本直接调用分类器，判断逻辑是否正确
- 用 `/health` 和日志确认服务是否正常
- 遇到接口超时或连接拒绝时，优先处理进程状态，不急着改分类逻辑

这个判断很关键，因为它避免了为了修复“服务不稳定”而去错误调整意图识别规则。

#### 5. 用调试脚本看“命中了什么关键词”

当出现“本地感觉应该对，但结果不对”时，没有继续猜，而是增加了调试脚本，直接打印：

- 关键词分类结果
- 每类关键词实际命中的内容
- 最终分类结果

这么做的目的，是把分类过程显式化。

例如“感冒一般怎样治疗？”这个例子，暴露出一个非常具体的问题：

- “感冒”容易把请求拉向 `symptom_consult`
- “怎样治疗”其实更接近 `medical_knowledge`

这个样例说明，单纯堆关键词不够，必须处理优先级和边界语义。因此后续调整了判断顺序，把“病历查询”“报告分析”“医学知识问题”放在更靠前的位置，再把典型症状咨询放后面。

#### 6. 每次只改一个变量，再回归测试

测试期间没有一次性大改分类器，而是遵循“小步调整、逐次验证”的方式：

1. 先确认失败案例
2. 只调整一个因素，例如提示词、关键词集合或优先级顺序
3. 先跑单例验证
4. 再跑批量验证

这样做的好处是，任何一次结果改善都能明确对应到具体改动，不会出现“改了很多东西，但不知道究竟哪一步生效”的情况。

#### 7. 特别关注会话记忆对分类的影响

因为项目已经接入 Redis + MySQL 的聊天记忆，所以测试 Router Agent 时，不能把它当成纯无状态分类器看待。

测试里专门注意了这一点：

- 批量测试尽量使用独立 `session_id`
- 避免前一轮对话内容影响后一轮分类
- 当接口结果与本地逻辑不一致时，优先怀疑上下文拼接是否带来了偏移

这一步的意义在于，Router Agent 实际接收到的 query，很多时候已经不是用户原始输入，而是“历史上下文 + 当前问题”的组合文本。测试必须按真实运行方式来考虑。

#### 8. 最后的判断标准不是“模型看起来聪明”，而是“路由是否稳定”

这次优化的验收标准并不是让 LLM 输出一段漂亮解释，而是更务实的三个指标：

- 意图是否落在四个允许值内
- 意图是否能稳定映射到正确 Agent
- LLM 不可用时，关键词回退是否仍然可用

也就是说，测试重点始终是“系统稳定性”和“路由可控性”，而不是单纯追求模型的自由发挥。

#### 9. 这次测试得到的结论

这轮测试最终验证了几件事：

- LLM 适合做复杂边界语义判断，但不能完全依赖它
- 关键词回退仍然是保障稳定性的必要机制
- 病历、报告、知识、症状这四类意图的边界，需要通过优先级明确约束
- 带会话记忆的接口测试，必须和本地逻辑测试分开看

换句话说，这次不是单纯“把分类做出来”，而是把 Router Agent 的测试方法也固定下来了：先局部、后整体；先单例、后批量；先排逻辑、再排链路。

### 关键改进

1. **混合分类策略**
   - LLM 用于复杂语义理解
   - 关键词回退用于快速、可靠的分类
   - 智能组合最大化准确率

2. **优化的关键词优先级**
   ```
   优先级1: 病历关键词 (历史、既往、以前)
   优先级2: 报告关键词 (检查、化验、指标)
   优先级3: 医学知识 (怎样、如何、治疗)
   优先级4: 症状关键词 (症状、发烧、咳嗽)
   ```

3. **配置扩展**
   - 新增 `ollama_base_url` 和 `ollama_model` 配置项
   - 支持灵活的 LLM 模型切换
